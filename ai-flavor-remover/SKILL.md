# AI 味去除 - 通用版

> 让 AI 生成的文字读起来像人写的

## 快速开始

```bash
# 使用默认模式（中度润色）
ai-flavor-remover <文章.md>

# 指定模式
ai-flavor-remover --mode=light <文章.md>
ai-flavor-remover --mode=heavy --domain=tech <文章.md>
```

## 核心设计：三层润色模式

| 模式 | 改动幅度 | 适用场景 | 耗时 |
|------|---------|---------|------|
| **light** 轻度 | 10-20% | 已经很自然，只需微调 | 快 |
| **medium** 中度 | 30-50% | 标准 AI 生成文本 | 中 |
| **heavy** 重度 | 60-80% | 机械感强，需大幅改写 | 慢 |

### 各模式具体策略

**light（轻度）- 精准手术**
- 替换高频 AI 词："值得注意的是"、"不难发现"、"基于以上分析"
- 打破机械句式："首先...其次...最后" → 隐性过渡
- 微调连接词："因此" → "所以/这样一来"

**medium（中度）- 全面重构**
- light 的全部操作
- 长短句重组：避免匀速叙述，加入节奏变化
- 注入情感标记：反问、感叹、口语化短句
- 视角个性化：适当加入第一人称或读者指向

**heavy（重度）- 彻底重写**
- medium 的全部操作
- 叙事结构重组：打破原有段落逻辑
- 风格定制：根据 domain 注入特定语体特征
- 细节具象化：抽象概念转具体场景

## 领域适配模板

通过 `--domain` 参数选择适配的写作风格：

### tech（技术文章）
- 保留专业术语，但解释要接地气
- 用类比和比喻降低认知门槛
- 允许适度的"硬核"表达，避免过度口语化
- 保留代码块和技术细节

**风格关键词**：精准、务实、有洞见、不炫技

### essay（随笔散文）
- 情感优先，逻辑次之
- 多用感官描写和场景还原
- 允许碎片化、意识流式的叙述
- 强调个人体验和独特视角

**风格关键词**：温度、细腻、留白、共鸣

### business（商业分析）
- 数据驱动，观点明确
- 用案例和故事包装抽象概念
- 结论前置，论证在后
- 适度使用行业术语建立权威感

**风格关键词**：洞察、锐利、有说服力、 actionable

### casual（ casual 内容）
- 最大程度口语化
- 允许网络用语和表情符号
- 短句为主，节奏轻快
- 像朋友聊天一样的平等视角

**风格关键词**：轻松、好玩、不做作、有梗

## AI 味检测清单

润色前自动扫描以下特征：

### 🔴 高频 AI 词（必改）
- [ ] 值得注意的是
- [ ] 不难发现
- [ ] 基于以上分析
- [ ] 综上所述
- [ ] 从某种程度上说
- [ ] 显而易见
- [ ] 换句话说

### 🟡 机械结构（建议改）
- [ ] "首先...其次...最后" 式并列
- [ ] 每段开头都是"XX 是..."
- [ ] 过度使用 bullet point
- [ ] 句子长度过于均匀
- [ ] 连接词重复使用

### 🟢 情感缺失（重度模式改）
- [ ] 全文无第一人称
- [ ] 无反问或感叹
- [ ] 缺乏具体场景/例子
- [ ] 读者指向模糊（"用户"vs"你"）

## 替换词库

### 高频词替换表

| AI 味词汇 | 替换选项 |
|----------|---------|
| 值得注意的是 | （删除）/ 有意思的是 / 你会发现 |
| 不难发现 | 明眼人都看得出 / 稍微留意就会发现 |
| 基于以上分析 | 说到底 / 往深里说 / 说白了 |
| 综上所述 | 总结一下 / 说到底 / 所以 |
| 从某种程度上说 | 某种程度上 / 某种程度上讲 |
| 显而易见 | 很显然 / 明摆着 / 不用说也明白 |
| 换句话说 | 说白了 / 也就是说 / 换言之 |
| 因此 | 所以 / 这样一来 / 结果就是 |
| 此外 | 还有 / 另外 / 再说 |
| 然而 | 但是 / 不过 / 问题是 |

### 过渡方式替换表

| 机械过渡 | 自然过渡 |
|---------|---------|
| 首先 | （直接开始）/ 先来说说 |
| 其次 | 再说 / 另外 / 还有一点 |
| 最后 | 最后说说 / 说到底 / 至于 |
| 一方面...另一方面 | 表面看...实际上... |

## 使用示例

### 示例 1：技术文档轻度润色

**原文：**
```
值得注意的是，OpenClaw 的 Gateway 层采用了微内核架构。
这种设计首先提高了系统的可扩展性，其次增强了安全性。
```

**light 模式：**
```
OpenClaw 的 Gateway 层采用了微内核架构。
这种设计提高了系统的可扩展性，也增强了安全性。
```

### 示例 2：产品介绍中度润色

**原文：**
```
我们的产品具有以下优势：
1. 高性能：处理速度提升 50%
2. 易用性：无需代码即可上手
3. 可扩展：支持自定义插件
```

**medium 模式 --domain=business：**
```
用起来什么感觉？三个字：快、简单、灵活。

处理速度比同类产品快一半，这是"快"。
不需要写一行代码就能跑起来，这是"简单"。
想要特殊功能？自己写个插件插上就行，这是"灵活"。

说白了，我们就是把复杂的东西藏在了底层，
留给你的只有顺手的体验。
```

### 示例 3：随笔重度润色

**原文：**
```
AI 技术正在改变我们的生活方式。
它在医疗、教育、交通等领域都有广泛应用。
这种变革带来了机遇，也带来了挑战。
```

**heavy 模式 --domain=essay：**
```
你有没有想过，上一次"亲手"做一件事是什么时候？

可能是查资料——现在问 AI 就行了。
可能是写邮件——现在 AI 能帮你起草。
可能是规划路线——AI 比你更懂哪条路不堵。

我妈前阵子去医院，CT 片子是 AI 先读了一遍，
标出可疑区域，医生再看。
效率高了吗？高了。但我妈出来问我：
"那医生是不是不用学那么多了？"

我一时语塞。

这就是我们现在的生活——便利得像蜜糖，
但你总忍不住想，蜜糖下面会不会有钩子。
```

## 高级配置

### 自定义词库

创建 `.ai-flavor-remover.json`：

```json
{
  "mode": "medium",
  "domain": "tech",
  "customReplacements": {
    "原词": "替换词",
    "系统": "平台"
  },
  "preservePatterns": [
    "API",
    "JSON",
    "正则表达式"
  ]
}
```

### 批量处理

```bash
# 处理整个目录
ai-flavor-remover --mode=light --batch ./posts/

# 处理并输出到新目录
ai-flavor-remover --mode=medium --output ./polished/ ./raw/
```

### 与 CI/CD 集成

```yaml
# .github/workflows/polish.yml
name: AI Flavor Remover
on:
  push:
    paths:
      - '_posts/**'
jobs:
  polish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run AI Flavor Remover
        uses: hylarucoder/ai-flavor-remover@v1
        with:
          mode: 'medium'
          domain: 'tech'
          path: '_posts/'
```

## 最佳实践

1. **先检测，后润色**
   ```bash
   ai-flavor-remover --detect-only <文章.md>
   ```

2. **分级处理**
   - 初稿用 `heavy` 彻底重写
   - 修订用 `medium` 精细调整
   - 终稿用 `light` 局部打磨

3. **领域保持一致**
   同一系列文章使用相同的 domain 设置，保持风格统一

4. **人工最终把关**
   AI 润色后务必通读一遍，确保没有偏离原意

## 局限与注意

- **无法替代人工**：重度模式可能改变原意，需人工复核
- **专业内容慎改**：学术论文、法律文件建议只用 light 模式
- **多语言支持**：目前主要针对中文优化

## 与 human-writing 的协同

这两个 Skill 是**互补关系**：

| Skill | 职责 | 作用时机 |
|-------|------|---------|
| **human-writing** | 从0到1写出好内容（心法+战略+技法） | 写作前/中 |
| **ai-flavor-remover** | 让好内容读起来更自然（润色+去机械化） | 写作后 |

**完整写作流程**：

```
Step 1: 选题
└─> human-writing --check --topic="xxx"
    └─> 确认选题有势能（专业+兴趣+时机）

Step 2: 框架
└─> human-writing --template=story
    └─> 按冲突→探索→发现→高潮→结局 搭建结构

Step 3: 写作
└─> 填充内容，保持活人感（第一人称、真实体验）

Step 4: 润色 ← 你在这里
└─> ai-flavor-remover --mode=medium --domain=tech
    └─> 去除"值得注意的是"等AI高频词
    └─> 长短句重组，注入口语化表达

Step 5: 质检
└─> human-writing --score article.md
    └─> HKR 评分（愉悦感/知识/共鸣）
```

**关键区别**：
- `human-writing` 解决"写什么"和"怎么写出灵魂"
- `ai-flavor-remover` 解决"怎么读起来不像机器写的"

两者配合，从选题到终稿，覆盖完整写作链路。
